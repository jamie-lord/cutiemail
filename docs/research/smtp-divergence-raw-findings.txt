[SEARCH] SMTP smuggling exploits END-OF-DATA confusion by tricking a sending MTA into forwarding a non-standard END-OF-DATA sequence <LF>.<LF> or <LF>.<CR><LF>, which a receiving MTA like Postfix interprets as the standard <CR><LF>.<CR><LF>, prematurely ending the message and letting attacker-controlled SMTP commands smuggle a spoofed second message.

[SEARCH] The Postfix SMTP-smuggling advisory assigns CVE-2023-51764 to Postfix, CVE-2023-51765 to Sendmail, and CVE-2023-51766 to Exim.

[SEARCH] Postfix's long-term fix (3.8.5+) adds smtpd_forbid_bare_newline, which with value 'normalize' processes a bare <LF> as <CR><LF> or 'reject' rejects it; it is disabled by default on Postfix versions before 3.9, with smtpd_forbid_bare_newline_exclusions defaulting to $mynetworks.

[SEARCH] The short-term Postfix mitigation is to set smtpd_data_restrictions = reject_unauth_pipelining and smtpd_discard_ehlo_keywords = chunking, silent-discard, blocking the pipelining and CHUNKING/BDAT vectors.

[SEARCH] Venema attributes the root cause to a decades-old Sendmail compatibility decision that accepted the non-standard bare <LF> line ending in addition to the standard <CR><LF>.

[SEARCH] RFC 5321bis (draft -44, the current SMTP standard-in-progress) mandates that CRLF (0x0D 0x0A) is the ONLY valid line terminator and that conforming implementations MUST NOT recognize or generate any other character or sequence as a line terminator — the exact clause SMTP smuggling exploits when MTAs disagree on bare LF/CR.

[SEARCH] The draft explicitly acknowledges bare CR or bare LF as a known problem source and requires SMTP clients to transmit CR/LF only as a CRLF sequence, but stops short of the line-ending strictness needed to close smuggling — it constrains the sender, not the receiver's parsing.

[SEARCH] The end-of-data indicator is specified as the exact byte sequence <CRLF>.<CRLF> (a line containing only a period, hex 2E), with a transparency (dot-stuffing) procedure — this is the §3.3/§4.5.2 primitive whose divergent interpretation across MTAs is the root of SMTP smuggling.

[SEARCH] As of draft -44 (2025-07-31) the receiver-side handling of bare CR/LF remains discretionary (MAY reject), not mandatory — meaning RFC 5321bis does NOT force receiving servers to reject the malformed line endings that enable smuggling, so the divergence is not closed at the spec level.

[SEARCH] The current SMTP standards-track draft is draft-ietf-emailcore-rfc5321bis-44, dated 31 July 2025, and its Security Considerations (Section 7) do not address SMTP smuggling or line-ending interpretation divergence between MTAs.

[SEARCH] The root cause was that Postfix (and similarly-architected servers) switched the TLS layer by replacing only the read/write functions while retaining the old stream input buffers, so plaintext data buffered before the handshake lingered and was consumed as if it were post-TLS input.

[SEARCH] The RFC mandates the end-of-data indicator be \r\n.\r\n (<CR><LF>.<CR><LF>), but OS-level line-break differences led implementations to also accept other formats such as \n.\n, and SMTP smuggling exploits this inconsistency between sending and receiving MTAs: the sending MTA must ignore the crafted end-of-data indicator embedded in the body while the receiving MTA misinterprets it, splitting one message into two and using the pipeline mechanism to process the smuggled second email.

[SEARCH] Measurement found 19 public email services, 1,577 private email services (of Tranco Top 10,000), five open-source email software packages, and one email gateway still vulnerable to SMTP smuggling as of the study. On the RECEIVING side 8 of 22 public services were vulnerable (Sina, Sohu, Yandex, Fastmail, cock.li, daum.net, rambler.ru, freemail.hu); open-source Haraka and Axigen plus the TrendMicro gateway were vulnerable on receiving; Postfix/Sendmail/Exim were vulnerable on the SENDING side (18 of 20 public services vulnerable sending).

[SEARCH] SMTP smuggling bypasses SPF and DMARC because the smuggled (second) email is delivered over the sending provider's authorized IP: SPF passes (the IP belongs to the spoofed domain's SPF record) and DMARC passes because SPF passed AND the From: header aligns with the MAIL FROM domain; this is a mandatory RFC workflow, not a configurable policy. Shared SPF infrastructure magnifies this: e.g. 63,225 Tranco Top-1M domains share Gmail's SPF record and 81,718 share Outlook's, so a free Gmail account can spoof any of those domains (demonstrated by spoofing admin@youtube.com to a Daum.net victim, passing SPF and DMARC).

[SEARCH] 13 of the tested sending-side services used dot-stuffing (adding an extra period to lines beginning with a period, altering \n.\n to \n..\n) as a mitigation, and some (Mail.ru, iCloud) used string filtering adding spaces around bare \n/\r; however these defenses often fail to filter the \x00 (NUL) character, so Gmail and Yahoo remained exploitable via payloads A7-A13 because they treat \x00 as a string terminator (common in C-based servers).
